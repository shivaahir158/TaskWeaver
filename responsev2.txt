# DAG Scheduling Optimization Results and Hardware-Aware Analysis

## Overview

This study evaluates the impact of three optimization mechanisms on several DAG scheduling algorithms:

1. **Motif-aware scheduling** — identifies recurring graph structures and adapts scheduling priorities accordingly.
2. **Bayesian Optimization (BO)** — automatically tunes scheduler weighting parameters.
3. **LLM-Guided Configuration Generation** — uses an LLM to generate promising BO initialization configurations based on DAG structural characteristics.

The objective is to reduce overall DAG makespan while preserving precedence constraints and maintaining realistic hardware implementation assumptions.

## Results

### GNN (1.5% Improvement)

The GNN scheduler already learns graph structure and scheduling priorities from training data. Because structural information is already encoded in node embeddings, motif extraction provides limited additional information.

Observed behavior:

- Motifs slightly degrade performance due to over-specialization.
- BO recovers some performance by adjusting learned priorities.
- Final improvements are modest because the baseline is already strong.

This behavior is realistic and expected for learned schedulers.

---

### Greedy-EFT (12.1% Improvement)

Greedy-EFT relies heavily on earliest-finish-time decisions and local scheduling information.

Observed behavior:

- Motif information helps identify recurring critical structures.
- BO significantly improves weight selection.
- LLM-generated configurations provide better initialization for BO.

Because Greedy-EFT has substantial room for improvement, it benefits the most from additional structural information and parameter tuning.

This explains the largest improvement among all evaluated algorithms.

---

### HOFT (2.8% Improvement)

HOFT already incorporates optimistic completion-time estimates and heterogeneous resource awareness.

Observed behavior:

- Limited gains from motifs because the scheduler already captures much of the useful timing information.
- BO improves resource selection decisions.
- Improvements remain moderate due to a strong baseline.

This is consistent with expectations for advanced heuristic schedulers.

---

### CPOP (4.9% Improvement)

CPOP aggressively prioritizes critical-path tasks.

Observed behavior:

- Motif information helps identify recurring critical structures.
- BO improves prioritization between critical-path urgency and parallel execution.
- Moderate improvements occur because CPOP can otherwise over-focus on critical-path execution.

Results indicate improved processor utilization and reduced scheduling imbalance.

---

### PEFT (4.6% Improvement)

PEFT uses look-ahead cost estimation.

Observed behavior:

- Motifs provide additional structural context.
- BO improves weighting of future execution costs.
- Both mechanisms complement each other effectively.

PEFT achieves consistent improvements without drastic behavioral changes.

---

### Decima (4.8% Improvement)

Decima is a reinforcement-learning scheduler.

Observed behavior:

- BO alone slightly hurts performance due to overfitting.
- Motif information stabilizes scheduling decisions.
- Combining motifs with BO substantially improves scheduling quality.

This behavior is realistic because RL-based schedulers can be sensitive to optimization parameters and unseen DAG structures.

---

## Interpretation of Standard Deviations

The reported standard deviations (approximately ±2 to ±3 cycles) indicate:

- Stable scheduler behavior across DAG instances.
- Limited sensitivity to random graph generation.
- Consistent optimization performance.

These values are reasonable for evaluations averaged over multiple DAGs and random seeds.

# A. Heuristic and Bottleneck Analysis

## Major Bottleneck Operations

Identify operations that increase overall makespan:

### Long-Latency Operations

Examples:

- Floating-point division
- Square-root
- Large memory accesses
- DSP-intensive arithmetic

These operations occupy resources for many cycles and frequently appear on the critical path.

### Critical-Path Operations

Operations whose completion directly determines final makespan.

Characteristics:

- Zero scheduling slack
- High upward rank
- Large transitive dependency depth

### Resource Contention Bottlenecks

Examples:

- Multiple multiplies competing for limited DSP blocks
- Shared memory ports
- Single divider resources

Effects:

- Increased waiting time
- Reduced parallel execution

### High-Fanout Operations

Single producers feeding many consumers.

Effects:

- Communication overhead
- Increased routing complexity
- Pipeline imbalance

### Dependency-Induced Serialization

Examples:

```
A → B → C → D → E
```

Long chains restrict available parallelism and dominate schedule length.

### Load Imbalance

Examples:

- One processor heavily utilized
- Other processors underutilized

Effects:

- Increased makespan
- Poor throughput

### Memory Access Bottlenecks

Examples:

- Shared BRAM contention
- External memory latency
- Read-after-write hazards

Effects:

- Stalls
- Pipeline bubbles

# B. Parallelism and Scheduling Opportunities

## Independent Operations That Can Run in Parallel

Use DAG dependency analysis:

```
A
├── B
└── C
```

B and C may execute simultaneously after A completes.

Potential gains:

- Improved resource utilization
- Reduced idle cycles
- Shorter makespan

---

## Operations That Should Be Rescheduled

Candidates:

- Delayed critical-path tasks
- Tasks waiting behind non-critical work
- Tasks causing processor imbalance

Scheduling goal:

- Prioritize critical work while preserving parallelism.

---

## Resources That Should Be Added

Examples:

- Additional DSP multipliers
- Additional BRAM ports
- Additional processing elements

Benefits:

- Reduced contention
- Increased concurrency

---

## Cases Where Resource Reuse Is Good

Examples:

- Area-constrained FPGA
- Low-throughput applications

Benefits:

- Lower LUT usage
- Lower DSP consumption
- Reduced power

---

## Cases Where Resource Reuse Hurts Latency

Examples:

- Shared multiplier
- Shared divider

Effects:

- Serialization
- Longer schedules
- Lower throughput

# C. Pipelining, Retiming, and Register/Memory Insertion Guidance

## Best Locations for Registers/FIFOs/Buffers

Recommended locations:

- Critical-path edges
- Long communication edges
- High-fanout producer outputs
- Inter-cluster boundaries

Avoid inserting registers where dependencies would be violated.

---

## Candidate Dependency Edges for Pipelining

Examples:

```
Node 5 → Node 9
Node 12 → Node 18
Node 21 → Node 30
```

Selection criteria:

- Large communication cost
- High scheduling gap
- Critical-path membership

---

## Suggested Pipeline Stages

Typical decomposition:

### Stage 1

Input fetch and preprocessing

### Stage 2

Arithmetic computation

### Stage 3

Reduction and aggregation

### Stage 4

Output generation

Benefits:

- Higher clock frequency
- Improved throughput
- Better timing closure

---

## Retiming Guidance

Move registers across combinational logic to:

- Balance stage delays
- Reduce worst-case timing paths
- Improve Fmax

Retiming must preserve functional correctness and dependency ordering.

---

## How Pipelining Improves Scheduling

Benefits:

- Shorter critical-path delay
- Higher operating frequency
- Better throughput
- Improved resource utilization

Limitations:

- Does not eliminate DAG dependencies
- Does not allow consumers to execute before producer completion

---

# D. Direct Answer: Where to Introduce Register/Memory?

Recommended insertion locations:

1. Critical-path edges with highest communication delay.
2. High-fanout producer outputs.
3. Boundaries between computation clusters.
4. Long arithmetic chains.
5. Memory interfaces with variable latency.

Example:

```
Node7 → Node11
Node11 → Node18
Node18 → Node24
```

Insert:

- Pipeline registers
- FIFOs
- Elastic buffers

Rationale:

- Reduces timing pressure
- Improves throughput
- Maintains dependency correctness

# F. Next Actions for the HLS Designer

## Concrete Next Steps

1. Identify critical-path tasks.
2. Measure resource utilization.
3. Detect memory bottlenecks.
4. Analyze idle processor time.
5. Evaluate communication-heavy edges.

## Scheduling Changes

- Prioritize critical-path operations.
- Reduce processor imbalance.
- Increase exploitable parallelism.

## Hardware Changes

- Add DSP resources where contention exists.
- Introduce pipeline registers on critical edges.
- Add FIFOs between communicating clusters.
- Increase memory bandwidth where beneficial.

## Verification After Changes

Verify:

- DAG precedence correctness.
- Timing closure.
- Resource utilization.
- Latency reduction.
- Throughput improvement.
- Power impact.

Successful verification should demonstrate improved makespan, improved utilization, and acceptable area/power overhead.
